# 작업 정리 — 주문 · 결제 · 배송비 · 관리자 주문관리

> 담당: 옥주 / 기준: `기능명세서 API 명세서/4_주문_결제_배송`, `.../7_관리자_공통`
> 모든 응답은 공통 래퍼 `CommonResponse<T>` 로 감싸고, 필드는 camelCase 로 프로젝트 규칙에 맞춤(명세의 snake_case는 예시).
> 결제 연동은 **포트원 V2**(api.portone.io) 사용. 검증 식별자는 `imp_uid` 가 아닌 **`paymentId`**(= 주문번호).

## ✅ 체크리스트 (완료)

### 04. 주문/결제
- [x] **주문 준비** — `POST /order/prepare`
  - 재고 확인(차감 X) → 주문 `PENDING_PAYMENT` 임시생성 + 결제 예정금액 확정 + 구매자 정보 반환
  - 배송지: `deliveryId`(주소록) 선택 또는 `shipping` 직접 입력
  - 쿠폰: `userCouponId` 선택 시 할인 재계산(본인/ACTIVE/유효기간/최소주문금액 검증)
- [x] **결제 검증 및 주문 확정** — `POST /order/complete`
  - 포트원 V2 로 실제 결제금액 조회 → DB 예정금액과 대조
  - 일치: 재고 차감 + 쿠폰 USED + 주문 `PAYMENT_COMPLETED` + 결제 `PAID` (트랜잭션 원자처리)
  - 불일치/미결제: 포트원 결제취소 호출 + 예외(롤백, 주문 PENDING 유지)
- [x] **주문내역 조회** — `GET /order/list` (최신순 페이징, 대표상품명/상품수)
- [x] **주문 상세 조회** — `GET /order/{orderId}` (상품·배송지·결제, 타인 접근 403)
- [x] **환불 요청** — `POST /order/{orderId}/refund` (결제완료/배송준비중에서만)

### 13. 관리자 주문 관리
- [x] **주문 리스트 조회** — `GET /admin/order` (주문번호/구매자 검색 + 상태 + 기간 + 페이징)
- [x] **주문 상세 조회** — `GET /admin/order/{id}` (paymentId·사용쿠폰·manualOverride 등 관리정보 포함)
- [x] **주문 상태 변경** — `PATCH /admin/order/{id}/status` (전이 검증, 취소 시 재고+쿠폰 복구)
- [x] **환불 처리** — `POST /admin/order/{id}/refund` (승인/거절)

### 부가 — 배송비 정책 (요청사항)
- [x] **배송비 정책 조회** — `GET /admin/shipping-policy`
- [x] **배송비 정책 변경** — `PATCH /admin/shipping-policy`
  - 기본값: 무료기준 20,000원 / 기본 배송비 0원 (DB `shipping_policy` 단일행, 관리자가 변경)

## 🔁 주문 상태 흐름

```
PENDING_PAYMENT ──결제확정──> PAYMENT_COMPLETED ──> PREPARING_SHIPMENT ──> SHIPPED ──> COMPLETED
                                     │                     │
                                     └───────> CANCELLED <─┘        (배송 전 취소: 재고·쿠폰 복구)

사용자 환불요청:  PAYMENT_COMPLETED | PREPARING_SHIPMENT ──> REFUND_REQUESTED
관리자 환불승인:  REFUND_REQUESTED ──> REFUND_COMPLETED (+ 결제 CANCELLED, 재고복구, 기간남은 쿠폰복구)
관리자 환불거절:  REFUND_REQUESTED ──> (요청 직전 상태로 복귀)
```

- `manualOverride`: 관리자가 상태 변경 API 로 바꾸면 `true` (추후 스케줄러 자동진행 대상에서 제외용)
- 배송 상세상태 `shipping.status`(READY/PREPARING/SHIPPING/DELIVERED), 결제상태 `payment.status`(READY/PAID/FAILED/CANCELLED) 별도 관리

## 🔌 포트원 V2 연동 (키 발급 · 설정 위치)

**구조: 우리 백엔드 → 포트원(api.portone.io) → 토스페이먼츠(PG).** 키가 2세트라 헷갈리므로 정리한다.

| 값 | 어디서 인증 | 우리 백엔드가 쓰나? | 특징 |
|---|---|---|---|
| **포트원 V2 API Secret** | 백엔드 → 포트원 | ✅ **이것만** | `test_` 접두어 **없는** 긴 문자열 |
| Store ID (`store-...`) | 프론트 SDK | ❌ (프론트) | 결제창 `requestPayment` |
| 채널 키 (`channel-key-...`) | 프론트 SDK | ❌ (프론트) | 결제창 `requestPayment` |
| 토스 키 (`test_sk_`, `test_ck_`) | 포트원 → 토스 | ❌ | 포트원 콘솔 채널에만 등록 |

**V2 API Secret 발급 (공식 경로)** — 사이트는 **`admin.portone.io`** (토스 사이트 아님!)
> 콘솔 → 결제 연동 → 연동 관리 → **식별코드·API Keys** → **V2 API** 탭 → **새로운 API Secret 발급** → 복사
> (같은 화면에서 **Store ID** `store-...` 도 복사)
> ⚠️ "API 개별 연동 키" 화면의 `test_sk_/test_ck_` 는 토스 키라 **아님**.

**넣는 위치**

| 파일 | 넣을 값 |
|---|---|
| `application-dev.properties` | `PORTONE_API_SECRET=<발급받은 V2 API Secret>` |
| `application-prod.properties` | `PORTONE_API_SECRET=${PORTONE_API_SECRET}` (운영은 환경변수) |
| `application.yml` | `custom.portone.api-secret` 로 자동 매핑 (`api-url` 기본 `https://api.portone.io`) |
| `test-config.properties` | 더미값(테스트는 PortOneClient Mock) |

**서버 검증 방식** (구현: `PortOneClientImpl`)
```
GET https://api.portone.io/payments/{paymentId}
Authorization: PortOne {PORTONE_API_SECRET}
```
- 응답 `amount.total` 과 DB 예정금액 비교, `status == "PAID"` 확인, 불일치 시 `POST /payments/{paymentId}/cancel`.
- 키 유효성 빠른 확인: 위 GET 을 임의 paymentId 로 호출 → **401=키 오류 / 404=키 정상**(결제건만 없음).

**전체 흐름**
```
prepare(응답 paymentId,amount) → 프론트 PortOne.requestPayment({storeId,channelKey,paymentId,totalAmount:amount,...})
 → 결제 성공 → 프론트가 paymentId 를 /order/complete 로 전달 → 서버가 위 GET 으로 금액 재검증 → 확정
```

### 🧾 브라우저 실결제 테스트 페이지

개발용 결제 테스트 페이지를 포함했다: **`src/main/resources/static/portone-test.html`**

## 🧪 Postman 테스트 가이드

**사전 준비**
- MySQL + Redis 기동, `dev` 프로파일로 앱 실행 (`base = http://localhost:8080`)
- 상품 데이터(재고>0), 일반 회원 1, 관리자 회원 1 필요
- 인증: `/member/login` → `data.accessToken` → 이후 요청 헤더 `Authorization: Bearer {accessToken}`

**시나리오 A — 사용자 주문 (결제창 없이 검증 가능한 부분)**
1. `POST /member/login` → accessToken 확보
2. `POST /order/prepare` → 응답 `paymentId`, `amount` 메모
3. `GET /order/list` → 방금 주문(PENDING) 확인
4. (결제 후) `POST /order/complete` — ⚠️ 아래 주의
5. `GET /order/{orderId}` → 상세 확인
6. `POST /order/{orderId}/refund` → REFUND_REQUESTED 확인

> ⚠️ **`complete` 는 실제 결제가 있어야 통과한다.** Postman만으로는 결제창을 못 띄우므로:
> - (권장) 프론트 결제 테스트로 `paymentId` 결제 완료 후 그 값으로 `complete` 호출
> - 결제 없이 `complete` 호출 시 포트원에 해당 결제가 없어 **502(OR008)/422** 가 정상 동작
> - 순수 API 검증이 필요하면 H2 통합테스트(아래) 결과 참고 — 포트원을 Mock 으로 대체해 end-to-end 검증 완료

**시나리오 B — 관리자 (관리자 accessToken 사용)**
1. `GET /admin/order` → 주문 리스트
2. `GET /admin/order/{id}` → 상세
3. `PATCH /admin/order/{id}/status` `{ "status": "PREPARING_SHIPMENT" }`
4. 사용자가 환불요청한 주문에 `POST /admin/order/{id}/refund` `{ "approve": true }`
5. `GET /admin/shipping-policy` → `PATCH` 로 baseFee 변경

**시나리오 C — 방어로직 빠른 확인**
- 재고보다 큰 수량으로 `prepare` → 422(OR005)
- 다른 사용자 토큰으로 남의 `GET /order/{id}` → 403(OR009)
- 일반 사용자 토큰으로 `/admin/order` → 403
- `COMPLETED` 주문을 `PREPARING_SHIPMENT` 로 `status` 변경 → 422(OR012)

## 🛠 트러블슈팅 / 설계 메모

### 1. 포트원 V1 → V2 전환
- 처음 V1(api.iamport.kr, imp_key/imp_secret, imp_uid)로 구현했으나, 콘솔이 V2 채널(채널 키)이라 **V2**(api.portone.io)로 전환
- 차이: 토큰 발급 불필요(`Authorization: PortOne {API_SECRET}`), 식별자 `imp_uid` → **`paymentId`(주문번호와 동일)**, 금액은 `amount.total`
- `complete` 요청 바디도 `{ impUid, merchantUid }` → `{ paymentId }` 로 단순화

### 1-1. (키 혼동 주의) 토스 키 ≠ 포트원 V2 API Secret
- 포트원 콘솔 채널 화면의 `test_sk_`/`test_ck_`/`보안 키`/`채널 키` 는 **토스페이먼츠 PG 키** — 백엔드가 쓰지 않음
- 백엔드가 쓰는 건 `admin.portone.io` → 식별코드·API Keys → **V2 API → "새로운 API Secret 발급"** 으로 만든 값(접두어 없음)
- 검증법: `GET api.portone.io/payments/{임의}` → **401=키 오류, 404=키 정상**. (실제 키로 404 확인 완료)

### 1-2. curl 한글 인코딩(개발 환경)
- Windows Git Bash 에서 curl `-d` 에 한글을 직접 넣으면 cp949 로 전송돼 `Invalid UTF-8 (0xb1)` 500 발생
- → 요청 바디를 **UTF-8 파일**로 저장 후 `--data-binary @file` 로 전송(코드 문제 아님). Postman 은 기본 UTF-8 이라 무관.

### 2. 검증 실패 시 롤백 정책
- `complete` 는 `@Transactional`. 금액 불일치/미결제 시 포트원 취소 호출 후 예외 → **롤백**
- 재고/쿠폰은 성공 시에만 건드리므로, 롤백돼도 되돌릴 상태가 없음(주문은 PENDING 유지). 명세의 "복구 로직 불필요" 방침과 일치

### 3. 배송비 정책을 DB + 관리자 API로
- 요청에 따라 상수 대신 `shipping_policy`(단일행) 로 관리 → 관리자 API 로 변경 가능
- H2 테스트(플라이웨이 off)엔 행이 없어, 서비스가 없으면 기본값(20,000/0)으로 생성/폴백

### 4. 환불 쿠폰 복구 = 기간 기준
- 취소/환불 시 쿠폰 `validUntil` 이 남아 있으면 ACTIVE 로 복구, 지났으면 복구 안 함

### 5. 타 도메인 침범 최소화
- 재고/쿠폰 복구가 필요하지만 다른 도메인 서비스는 미구현 상태 → 주문 서비스 내부에서 엔티티 메서드로 처리
- 쿠폰 복구용 `UserCoupon.restore()` 만 추가(순수 신규 메서드)

### 6. 관리자 리스트 구매자 검색
- 주문 테이블엔 `memberId` 만 있어 이메일/이름 조인 불가 → 회원 검색으로 id 목록을 뽑아 주문 쿼리 IN 절에 전달(IN 빈 리스트 방지용 sentinel `-1`)
